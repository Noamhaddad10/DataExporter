## 2. Bugs potentiels

### 2.1 Patterns dangereux

| Fichier:ligne | Description | Gravite |
|---|---|---|
| `LoggerPduProcessor.py:133-135` | `except Exception as e:` suivi de `print()` au lieu de `log.error()` -- erreur silencieuse dans l'interpretation des EventReport | Moyenne |
| `LoggerPduProcessor.py:269` | `# TODO not a good fix` -- le developpeur reconnait que le code de marking_text est fragile | Faible |
| `LoggerSQLExporter.py:97` | `print(f"No data in {self.table_name}")` -- print de debug laisse dans le hot path, genere du bruit en production | Moyenne |
| `LoggerSQLExporter.py:116` | `print("count_export: ", count_export)` -- print de debug dans le hot path d'export | Moyenne |
| `kafka_main.py:135` | `except Exception as exc:` dans `check_sql_connection` -- trop large, masque les erreurs inattendues | Faible |

**`log.error()` sans `exc_info=True`** : la majorite des appels `log.error()` ne passent pas `exc_info=True`. Le traceback est perdu sauf quand `traceback.format_exc()` est passe manuellement (ce qui est fait dans `kafka_consumer.py:236` et `:404`). Les autres appels perdent le contexte d'erreur.

### 2.2 Gestion Kafka

OK globalement :
- `enable.auto.commit = False` dans `kafka_config.py:132` -- correct
- `consumer.commit(asynchronous=False)` dans `kafka_consumer.py:420` -- APRES SQL insert -- correct
- `producer.poll(0)` appele dans la boucle principale (`kafka_producer.py:141`) -- correct
- `producer.flush(timeout=30)` dans le finally -- correct
- Gestion `BufferError` via `_produce_with_retry()` avec 3 tentatives -- correct

PROBLEME MINEUR : `kafka_consumer.py:157-158` -- sur message trop court, on commit l'offset quand meme. C'est correct (message malformed, on ne veut pas le retraiter), mais merite un commentaire explicite.

### 2.3 Gestion SQL

CRITIQUE -- **Injection SQL** dans `kafka_main.py:182-184` :
```python
query = (
    f"SELECT TOP 1 LoggerFile FROM Loggers "
    f"WHERE LoggerFile LIKE '{logger}%' ORDER BY WorldTime DESC"
)
```
La variable `logger` provient de `cfg.LOGGER_FILE` (fichier de config JSON). Le risque est faible car la source est locale, mais le pattern est dangereux : si le JSON est corrompu ou modifie par un tiers, c'est une porte d'entree pour l'injection SQL. Devrait utiliser une requete parametree.

- Schema `dis` utilise dans `LoggerSQLExporter.py:199` (`sqlalchemy.MetaData(schema="dis")`) -- correct
- Transactions : `engine.begin()` utilise dans `Exporter.insert()` -- commit implicite en sortie du context manager -- correct
- Pas d'injection SQL ailleurs (SQLAlchemy ORM est utilise pour les inserts)

**Mutable default argument** dans `LoggerSQLExporter.py:51` :
```python
def __init__(self, ..., data=[]):
```
Bug classique Python : toutes les instances creees sans passer `data` partagent la **meme liste**. En pratique, ca ne cause pas de bug visible car `data` est ecrase immediatement dans la plupart des cas, mais c'est un piege latent.

### 2.4 Ressources non liberees

- `kafka_producer.py` : socket ferme dans `finally` (`sock.close()` ligne 191) -- OK
- `kafka_consumer.py` : consumer ferme dans `finally` (`consumer.close()` ligne 318) -- OK
- `kafka_main.py:60-61` : le `FileHandler` cree dans `start_log_listener()` n'est jamais ferme explicitement. Le `QueueListener.stop()` appele en fin de programme devrait s'en charger, mais ce n'est pas garanti.
- `kafka_main.py:127-128` : l'engine SQLAlchemy cree dans `check_sql_connection()` n'est pas dispose. Il reste en memoire avec son pool de connexions. Pas critique car c'est un check au demarrage, mais pas propre.
- `kafka_main.py:153-156` : meme probleme pour `_sql_conn_context()` -- cree un engine qui n'est jamais dispose.

## 6. Recommandations priorisees

### CRITIQUE (a corriger avant toute nouvelle feature)

1. **Injection SQL** -- `kafka_main.py:182-184` -- La variable `logger` est interpolee dans une requete SQL via f-string. Utiliser une requete parametree (`sqlalchemy.text("... WHERE LoggerFile LIKE :pattern")`, `{"pattern": f"{logger}%"}`).

2. **Mutable default argument** -- `LoggerSQLExporter.py:51` -- `data=[]` dans le constructeur de `Exporter`. Remplacer par `data=None` et `data = data or []` dans le body.

3. **Absence de requirements.txt** -- Les dependances (`confluent_kafka`, `sqlalchemy`, `pyodbc`, `pandas`, `opendis`) ne sont pas declarees. Creer un `requirements.txt` pour garantir la reproductibilite.

### IMPORTANT (a corriger dans les prochains sprints)

1. **Prints de debug dans le hot path** -- `LoggerSQLExporter.py:97` (`"No data in..."`) et `:116` (`"count_export: ..."`) generent du bruit continu en production. Remplacer par `log.debug()`.

2. **log.error() sans exc_info** -- Dans `kafka_consumer.py`, `kafka_main.py`, `LoggerSQLExporter.py` -- la plupart des `log.error()` ne passent pas `exc_info=True`, le traceback est perdu.

3. **Imports inutilises** -- 7 imports morts identifies (voir section 5.1). Nettoyage rapide.

4. **Variables globales mutables** -- `LoggerSQLExporter.py:17-19` (`count`, `count_export`, `count_insert`) et `LoggerPduProcessor.py:14` (`count_parsing`) sont des variables globales mutables utilisees comme compteurs. Probleme potentiel en multiprocessing (chaque process a sa propre copie, les compteurs sont faux).

5. **SQLAlchemy engines non disposes** -- `kafka_main.py:127`, `:153` -- les engines crees pour les checks de prerequis ne sont jamais fermes.

### AMELIORATION (quand tu auras le temps)

1. **Ajouter `.venv/` et `output/` au `.gitignore`** -- Eviter que les fichiers de sortie et le virtualenv ne soient accidentellement commites.

2. **Remplacer `pd.read_sql_query` par SQLAlchemy pur** dans `kafka_main.py:185` -- `pandas` est importe uniquement pour cette requete. Utiliser `conn.execute()` directement eliminerait la dependance `pandas` du module principal.

3. **Installer pytest** et adapter le framework de test pour permettre `pytest --cov` et l'integration CI.

4. **Docstrings manquantes** sur `LoggerPduProcessor.py` et `LoggerSQLExporter.py` au niveau module.

5. **Lignes > 120 chars** dans `LoggerPduProcessor.py` -- refactoring des longues concatenations DIS.
