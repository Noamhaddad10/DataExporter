### 2.2 Gestion Kafka

✅ **Côté consumer** (`kafka_consumer.py`) :
- `enable.auto.commit: False` en dur dans `kafka_config.py:133`.
- `_safe_commit()` appelé **après** `_drain_and_export()` si `sql_ok` est vrai (`kafka_consumer.py:252-267`).
- Pas de double commit : chaque branche `continue` appelle `_safe_commit()` une seule fois.
- `poll(timeout=1.0)` — bonne valeur pour réagir à `stop_event` en < 1 s.

✅ **Côté producer** (`kafka_producer.py`) :
- `producer.poll(0)` appelé à chaque message (ligne 140).
- `producer.flush(timeout=30)` dans le `finally` (ligne 185).
- `BufferError` géré avec retry + `poll(1)` (`_produce_with_retry`, lignes 255-272).

⚠️ **Observation** : `_safe_commit` fait un `commit()` (sans offset explicite) sur le dernier message `poll()` reçu. En cas de pluralité de partitions assignées à un même consumer, cela commit l'offset de tous les pointeurs internes, ce qui est le comportement correct — mais le code ne commit pas spécifiquement `TopicPartition(msg.topic(), msg.partition(), msg.offset()+1)`. Si on cherchait un commit au niveau message, il faudrait le faire explicitement. Pour un pipeline at-least-once « simple », c'est acceptable.

### 2.3 Gestion SQL

✅ **Bien fait** :
- Engine SQL créé par process (pas partagé) : `LoggerSQLExporter.__init__` instancie son propre `sql_engine`.
- Schéma `dis` utilisé correctement via `sqlalchemy.MetaData(schema="dis")` (`LoggerSQLExporter.py:196`), schéma `dbo` pour la table `Loggers` (`LoggerSQLExporter.py:232`).
- Pas de f-string dans les requêtes SQL utilisateur — `kafka_main.py:183-187` utilise `sqlalchemy.text(...)` + `params={...}` correctement.
- `Exporter.insert()` a une logique de retry propre (5 tentatives, back-off 50 ms sur deadlock, pas de rollback manuel).

🔴 **Bug comportemental — dead code** : `LoggerSQLExporter.py:99-115` (Exporter.export, branche EntityLocations) :

```python
elif self.table_name == "EntityLocations":
    if len(data) < self.export_size and self.count > 0:  # jamais vrai
        self.count += 1
        ...
    else:
        self.count = 0   # remet à 0 à chaque passage
```

`self.count` est initialisé à 0 dans `__init__`. À la première entrée, `0 > 0` est faux → on passe dans `else` → `self.count = 0`. La condition ne peut donc *jamais* passer `True`. Conséquence : la micro-optimisation « attendre `export_size` petits lots avant d'insérer » est totalement inopérante. Chaque batch part directement en SQL. Soit c'est voulu (alors supprimer la branche morte), soit la condition devrait être `len(data) < self.export_size` tout court et `self.count` a été ajouté par erreur.

🟡 **Architecture** : `Exporter.__init__` démarre une `threading.Thread` **non-daemon** (ligne 72). Si le propriétaire du thread ne `.set()` jamais son `stop_event`, le thread tourne à jamais via `threading.Timer(1, self.export)`. Voir §2.4.

### 2.4 Ressources non libérées

🔴 **CRITIQUE — `register_logger_in_db`** (`kafka_main.py:231-257` + `LoggerSQLExporter.py:206-219`) :

`kafka_main.main()` crée `stop_writing_event_main = multiprocessing.Event()` puis appelle `register_logger_in_db` qui instancie un `LoggerSQLExporter`. Ce constructeur crée inconditionnellement **4 `Exporter`** (`EntityLocations`, `Entities`, `FirePdu`, `DetonationPdu`), chacun démarrant une thread **non-daemon** bouclant via `threading.Timer(1, self.export)` aussi longtemps que `stop_event.is_set()` est faux.

`stop_writing_event_main` n'est `.set()` nulle part dans `kafka_main.py`. Lors du `Ctrl+C`, `main()` atteint sa fin logique mais :
- 4 threads non-daemon planifient un nouveau Timer toutes les secondes.
- Python attend la fin des threads non-daemon avant de terminer l'interpréteur.

**Conséquence** : `kafka_main.py` peut rester actif indéfiniment après le message `Pipeline stopped gracefully`. L'utilisateur doit forcer (double Ctrl+C / kill). Ce comportement n'est pas masqué par les consumers car leurs propres `stop_writing_event` **locaux** sont correctement `.set()` dans leur `finally` (`kafka_consumer.py:311`).

De plus, `lse.sql_engine` n'est jamais `dispose()` (20 connexions du pool restent ouvertes). Idem dans chaque consumer : `consumer.close()` est fait mais pas `lse.sql_engine.dispose()`.

🟡 **Autre** : `LoggerSQLExporter.insert_sync` pour `Loggers` crée un nouvel `Exporter` à chaque appel (`LoggerSQLExporter.py:250-256`) → un nouveau thread non-daemon à chaque écriture `Loggers`. Peu fréquent en pratique (appelé 1× au démarrage) mais patron discutable.

## 6. Recommandations priorisées

### 🔴 Critique (à corriger avant toute nouvelle feature)

1. **`kafka_main.py:316-322`** — Lever le `stop_writing_event_main` dans la séquence de shutdown, ou utiliser un Event déjà managé par le pipeline. Sans cela, `main()` ne termine pas proprement. *Pourquoi critique* : tout shutdown laisse le processus principal zombifié — symptôme visible par l'opérateur.

2. **`LoggerSQLExporter.py:99-115`** — Corriger la branche EntityLocations. Soit supprimer le code mort, soit remplacer la condition par `if len(data) < self.export_size:` sans dépendre de `self.count`. *Pourquoi critique* : comportement de batching documenté mais pas implémenté — écart code ↔ intention qui trompera le prochain dev.

### 🟡 Important (à corriger dans les prochains sprints)

3. **`kafka_config.py:53`** — Remplacer par `[t for t in _RAW["tracked_tables"].replace(" ", "").split(",") if t]` pour obtenir `[]` quand la clé vaut `""`. Impact actuel : `if table in tracked_tables` fonctionne par chance (table_name jamais `""`), mais le test `'' in [...]` retournera `False` au lieu de `True` par inadvertance.

4. **Libération des ressources SQL** — Dans `kafka_consumer.py` `finally` (ligne 304+) et dans `kafka_main.main()` après shutdown : appeler `lse.sql_engine.dispose()`. Évite les connexions pool orphelines côté SQL Server.

5. **Refacto de `run_consumer`** (`kafka_consumer.py:45`, 277 lignes) — Extraire le traitement d'un message en sous-fonctions. Couverture et tests unitaires deviendraient possibles.

6. **Infrastructure de tests pytest** — Convertir `test_deep.py`/`test_intensive.py`/`test_kafka_pipeline.py` en fonctions `test_*` avec marqueurs `@pytest.mark.integration` pour pouvoir lancer la partie unitaire sans broker, et ajouter `pytest` + `pytest-cov` à `requirements.txt`.
