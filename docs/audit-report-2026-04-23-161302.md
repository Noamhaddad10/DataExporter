**Top 3 des points positifs** :
1. **Bugs runtime corrigés et mesurés** — `socket.timeout` intercepté avant `OSError` (0 log/s idle confirmé), partitioning par `key=entity_id` ou round-robin (4 consumers actifs confirmés). Baseline 258 msg/s → 935 msg/s (x3.6).
2. **Auto-validation** — `/ship` exécute maintenant `/smoke-test` en phase 6/9, ce qui garantit que toute future PR sur le chemin chaud sera validée empiriquement avant merge.
3. **Config centralisée, imports propres, docstrings partout, 0 `except:` nu dans les modules actifs, `exc_info=True` partout où il faut**.

### 2.5 Patterns runtime (section 3.5 nouvelle)

**Première exécution de la section 3.5** — patterns que les audits précédents ne cherchaient pas.

#### 2.5.1 `TimeoutError` vs `except OSError` (3.5.1)
✅ **OK** — `kafka_producer.py:164-167` contient `except socket.timeout: continue` avant le `except OSError as exc` (ligne 168). Ordre correct. Preuve runtime : baseline post-fix 2026-04-23 → 0 log d'ERROR pendant 15 s d'idle. Pas de consumer concerné (le consumer n'a pas de recvfrom avec settimeout).

#### 2.5.2 Partitioning Kafka (3.5.2)
✅ **OK** — `kafka_producer.py:139-144` :

```python
pdu_type: int = received_data[2]
if pdu_type == 1 and len(received_data) >= 18:
    key: bytes | None = received_data[12:18]
else:
    key = None
```

Et `_produce_with_retry` passe `key=key, partition=<non spécifié>` → Kafka choisit : hash(key) si key, sinon round-robin sticky. Preuve runtime : baseline post-fix 2026-04-23 → distribution 206/237/263/294 sur 1000 FirePdus (4 consumers actifs).

#### 2.5.3 Chemins chauds avec appels réseau par message (3.5.3)
🟡 **Signalé** — goulot connu :
- `kafka_consumer.py:422` — `consumer.commit(asynchronous=False)` appelé dans `_safe_commit` pour chaque message traité (y compris skipped).
- `LoggerSQLExporter.py:290` — `self.exporters[table].insert(rows)` appelé par `insert_sync` pour chaque message.

Débit max théorique ≈ 1 / (latence_SQL + latence_commit_Kafka). Observé : **935 msg/s end-to-end** (baseline post-fix 2026-04-23). Objectif 20k msg/s = **×21 à trouver** via batching. C'est le sujet de l'issue perf à venir.

#### 2.5.4 Chemin async/sync dual (3.5.4)
🟡 **Dette architecturale confirmée** — `LoggerSQLExporter` expose deux systèmes :
- **Async** : `export()` (ligne 241) → `Exporter.add_data()` (ligne 93) → Timer → `Exporter.export()` (ligne 102) → `Exporter.insert()` (ligne 144)
- **Sync** : `insert_sync()` (ligne 258) → `Exporter.insert()` directement

Utilisations réelles :
- `kafka_consumer.py:397` → `lse.insert_sync()` **uniquement** (chemin chaud)
- `kafka_main.py:256` → `lse.export("Loggers", ...)` **une seule fois** au démarrage (mais pour `Loggers`, `Exporter.__init__` fait `self.insert(data)` direct sans passer par `add_data` — voir `LoggerSQLExporter.py:88-90`)

**Conclusion** : `add_data()` n'est appelé **nulle part en runtime réel**. Les Timer threads des 4 starter Exporters (`EntityLocations`, `Entities`, `FirePdu`, `DetonationPdu`) tournent en boucle vide, vérifient `self.data == []`, re-schedulent un Timer toutes les secondes. Coût : N_consumers × 4 threads = 16 threads qui ne font rien. Plus la complexité du `close()` qui doit gérer leur arrêt via `stop_event.set()` + `time.sleep(2.0)`.

**Recommandation** : supprimer `export()` / `add_data()` / branchement Timer dans `Exporter` si on confirme qu'aucun chemin de production n'en a besoin. Gain : simplification du shutdown, réduction de ~80 lignes, plus de fil non-daemon à gérer.

1. **Dette architecturale async/sync** (section 2.5.4) — supprimer `Exporter.export()`, `Exporter.add_data()`, et les chemins Timer associés. Gardé seulement `Exporter.insert()` (utilisé par `insert_sync()`). Gain direct : ~80 lignes supprimées, 16 threads de moins, shutdown plus simple, `close()` peut passer de `sleep(2.0)` à un simple `dispose()`.

2. **Batching SQL pour lever le plafond à 1000 msg/s** (section 2.5.3) — actuellement `insert_sync` fait 1 `INSERT ... VALUES (...)` par message. Passer à un buffer + `executemany` toutes les N rows ou toutes les M ms. Gain attendu : ×10-20. Combiné avec batch commit Kafka : ×30+ possible.

3. **Migrer les tests vers pytest** — créer `conftest.py` avec fixtures `engine` et `logger_file`, renommer les fonctions existantes pour match la signature pytest standard. Ajouter `pytest` et `pytest-cov` à `requirements.txt`. Permettra d'exécuter `pytest -m unit` / `-m integration`.

## 7. Méthodologie de correction suggérée

Ordre recommandé :

1. `/ship "supprimer Exporter.export() et Timer threads -- chemin mort dans le flow Kafka"` — préparation technique, shutdown simplifié (début de #1).
2. `/ship "batch SQL insert dans kafka_consumer -- accumuler N messages avant executemany"` — gros gain perf (#2).
3. `/ship "batch Kafka commits -- commit tous les K messages au lieu de chaque"` — deuxième gros gain perf (complément #2).
4. Ensuite seulement : `/smoke-test full` pour mesurer le nouveau débit.
5. Refacto `run_consumer` et migration pytest → après stabilisation perf.

Chaque `/ship` exécutera automatiquement `/smoke-test` en phase 6/9 et bloquera le merge si le pipeline casse.

## 8. Ce qui n'a PAS pu être audité (et comment le compléter)

- **Comportement du shutdown gracieux en console interactive** : `TaskStop` = kill brutal sur Windows. Nécessite un Ctrl+C humain dans un vrai terminal. Statut de la PR #5 sur ce point : non invalidé mais non démontré.
- **Débit 20k msg/s** : non atteint (935 msg/s mesurés 2026-04-23). Faisable si les recommandations 🟡 #1 et #2 sont mises en œuvre.
- **Charge mixte multi-types** : le smoke test 2026-04-23 a testé FirePdu uniquement. Un test avec EntityStatePdu + FirePdu + DetonationPdu simultanés validerait empiriquement la cache-coherence du nouveau partitioning par `key=entity_id`.
- **Cohérence avec le schéma SQL réel** : les 6 tables `dis.*` sont présumées correctes (présence vérifiée, structure non auditée).
- **Aucune CI GitHub Actions** : `.github/workflows/` absent. Pas d'automatisation de test/lint à l'ouverture de PR. Proposition : ajouter un workflow qui lance `pytest` (après migration) + `pyflakes` sur tout push.

**Recommandation explicite** : pour compléter cet audit, lancer `/smoke-test full` pour obtenir un débit de référence à jour.
